## 📝 2. 결과 리포트 (실패 원인 분석 및 시간 복잡도 분석)

본 시뮬레이터는 시스템 예외 처리(Exception Handling) 및 방어 로직의 견고함을 증명하기 위해, 4개의 의도적인 불량 데이터를 주입하여 테스트를 진행함. 프로그램 중단(Crash) 없이 **4개의 명시적 FAIL 케이스를 안전하게 격리 및 집계**하는 데 성공함. 실패 원인 분석은 다음과 같음.

* **첫째, 동점 처리(UNDECIDED) 방어 확인 (`size_3_undecided`)**
  모든 픽셀이 1로 채워진 패턴 입력 시, Cross 필터와 X 필터의 MAC 연산 결과가 완벽하게 일치함. 억지로 특정 라벨을 편향되게 판독하는 대신, `abs(A - B) < 1e-9` 정책에 따라 'UNDECIDED(판정 불가)'를 선언하여 판독 신뢰성을 증명함.
* **둘째, 스키마 누락 방어 확인 (`size_7_missing`)**
  JSON 딕셔너리에 존재하지 않는 `size_7` 필터를 요구하는 데이터 유입 시, `KeyError`로 인한 시스템 강제 종료를 방지하고 해당 케이스만 '필터 누락' 사유로 격리(Pass) 처리함.
* **셋째, 배열 크기 불일치 방어 (`size_3_mismatch`)**
  3x3 필터를 요구하면서 4x4 패턴을 넘겨주는 악의적 입력에 대해, 이중 루프 연산 전 배열 길이를 사전 검증하여 `IndexError`를 원천 차단함.
* **넷째, 정답 오기입(False Label) 판독 (`size_3_wrong`)**
  명백한 Cross 형태의 입력임에도 정답지(expected)가 'x'로 오기입된 상황에서, AI 엔진은 정상적으로 Cross 점수를 더 높게 산출함. 인간의 오기와 AI의 판독 불일치를 정확히 식별하여 FAIL 처리함.

**[시간 복잡도 분석 (O(N²) 증명)]**
시간 복잡도 측면에서 본 MAC 연산 엔진은 **$O(N^2)$**의 복잡도를 가짐. 입력된 $N \times N$ 크기의 2차원 배열 처리를 위해 행과 열을 순회하는 이중 `for` 루프를 사용하기 때문임. [모드 3] 성능 측정 결과에서 이 이론적 배경이 수치로 명확히 증명됨. 행렬 크기(N)가 5에서 25로 5배 증가 시, 실제 픽셀 연산 횟수($N^2$)는 25에서 625로 정확히 25배 폭증하며, 측정된 연산 시간(ms) 역시 N의 제곱 비율에 비례하여 폭발적으로 증가함. 이는 대규모 데이터(수백만 픽셀 이미지)를 단일 스레드로 직렬 처리할 때 심각한 병목 현상이 발생함을 시사하며, 대규모 행렬 연산을 동시 병렬 처리하는 **NPU(Neural Processing Unit)** 하드웨어 가속의 필수성을 뒷받침함.

## 🧠 4. 심층 분석 리포트 (평가기준 9~19 대응)

### Q1. MAC 연산 로직의 전체 흐름은 어떻게 설계되었는가? (기준 10)
1. **입력:** $N \times N$ 크기의 Filter와 Pattern 2차원 배열 로드
2. **탐색:** 이중 `for` 루프(`i`, `j`)를 사용하여 컴퓨터 메모리 캐시 지역성(Cache Locality) 최적화에 유리한 행 우선(Row-major) 방식으로 탐색
3. **곱셈 및 합산:** `filter_matrix[i][j] * pattern_matrix[i][j]` 연산 수행 후, 해당 결과값을 `total_score` 변수에 누적(Accumulate)
4. **반환:** 최종 산출된 float 스칼라 값을 반환

### Q2. 패턴 크기 추출 및 매칭 로직의 설계 방식은? (기준 11)
`size_13_2`와 같은 키에서 기준 필터 키를 분리하기 위해 `split('_')` 함수를 사용함. 인덱스 `[0]`과 `[1]`을 결합하여 `size_13`이라는 표준 키를 생성하고, 이를 `filters` 딕셔너리에 매핑하여 정답지 필터를 동적 로드함.

### Q3. 라벨 정규화 함수를 분리한 이유와 장점은? (기준 12)
데이터 전처리와 비즈니스 로직(MAC 판독)의 결합도를 낮추기 위함임. 사용자가 `+`, `x`, `Cross` 등 불규칙한 데이터를 입력하더라도 `normalize_label()`을 거쳐 표준 라벨로 강제 전처리됨. 결과적으로 메인 비교 로직이 단순화되며, 신규 라벨 규격 추가 시 파서 내부만 수정하면 되므로 확장성이 극대화됨.

### Q4. 시간 측정 시 I/O를 배제한 연산 구간 설정 이유는? (기준 13)
`print()`, `json.load()` 등 I/O 연산은 OS 스케줄링이나 디스크 상태에 따라 불규칙한 지연(Noise)을 발생시킴. 순수 알고리즘 복잡도($O(N^2)$) 기반 연산 성능만 정밀 측정하기 위해 고해상도 타이머(`perf_counter`)를 `calculate_mac` 전후에 밀착 배치함.

### Q5. MAC 점수와 유사도 간의 상관관계 및 판단 근거는? (기준 14)
필터 배열 내 타겟 픽셀(1.0)과 패턴 배열 내 픽셀(1.0)의 위치가 겹칠수록 곱셈 결과가 1로 보존되어 누적 점수가 상승함. 반면 위치가 어긋나면 0이 곱해져 점수가 누락됨. 즉, MAC 누적 점수의 상승은 두 배열의 픽셀 기하학적 형태가 유사함을 수학적으로 의미함.

### Q6. 연산 횟수와 측정 결과가 O(N²) 경향을 보이는 핵심 원리는? (기준 15)
NPU 엔진은 2차원 배열 전체 픽셀을 순차 접근하기 위해 이중 루프 구문을 사용하기 때문임. 실제 출력된 [모드 3] 성능 표 데이터에 따르면, N 크기가 5배 커질 때 연산 횟수($N^2$) 및 실제 측정 소요 시간 모두 제곱에 비례하여 약 25배 폭증함. 이는 이중 루프에 기인한 $O(N^2)$ 복잡도를 증명함.

### Q7. 부동소수점 오차 발생 원인과 비교 정책(epsilon)의 필요성은? (기준 16)
컴퓨터의 실수 처리 로직(이진수 분수 변환) 특성상 무한소수 처리 시 미세한 절삭 오차가 필연적으로 발생함. 수학적으로 동일한 값이어도 `1.0`과 `0.999999999`처럼 다르게 산출될 수 있음. 단순 `==` 비교 시 오답 처리가 유발되므로, `abs(A - B) < 1e-9` 구조의 허용 오차(Epsilon) 비교 정책 도입이 필수적임.

### Q8. 모드별 동점 처리 규칙(UNDECIDED)을 다르게 설정한 이유는? (기준 17)
AI 모델 판독의 신뢰성 방어를 위함임. 오차 범위 내의 득점 차이 발생 시 시스템이 강제로 1순위 라벨을 채택할 경우(False Positive), 실무에서 심각한 오류 판단을 초래할 수 있음. 이에 따라 동점 확인 시 무리한 결정을 배제하고 "판독 불가(UNDECIDED)" 상태를 명시적으로 반환하도록 설계함.

### Q9. 새로운 라벨(예: Circle) 추가 시의 확장 프로세스는? (기준 18)
1. **데이터 확장**: `data.json`의 `filters` 내에 `"Circle"` 2차원 배열을 추가.
2. **파서 확장**: `normalize_label()` 내부에 `'o'`, `'circle'` 입력을 `"Circle"`로 치환하는 분기문 추가.
3. **로직 리팩토링**: `compare_scores()` 함수를 다자간 비교 딕셔너리로 수정하고 `max()` 탐색을 통해 최댓값을 산출하되, 1-2위 간 편차가 epsilon 미만일 경우 `UNDECIDED` 처리하도록 재설계.

### Q10. 대규모 패턴 처리 시 문제점과 최적화 방안, CPU/NPU 병목은? (기준 19)
Python의 단일 스레드 구조는 수백만 픽셀 이미지 연산 시 $O(N^2)$ 복잡도와 맞물려 극심한 순차 연산 병목 현상을 유발함. 최적화를 위해선 2차원 배열을 1차원으로 Flatten하여 캐시 적중률을 높이거나, 근본적으로 수천 개의 MAC 코어를 내장하여 거대한 행렬 연산을 단일 사이클에 병렬 처리하는 NPU(Neural Processing Unit) 아키텍처 도입이 불가피함.
